# Repository Rotation System

To cultivate a rich, diverse engineering profile, activity must rotate between different scopes of work. Committing to only one repository signals a lack of ecosystem maturity.

## The Rotation Cycle (Bi-Weekly)

### Week A: The Core Focus
*   **70% Effort:** `dev-utility-lab` Flagship Maintenance.
    *   Address technical debt, refactor old functions, optimize CI patterns.
*   **20% Effort:** External OSS Outbound Contributions.
    *   Submit 2 small documentation/test pipeline PRs to target external repos.
*   **10% Effort:** Community Engagement & Discussions.
    *   Answer questions in targeted GitHub discussion forums.

### Week B: Ecosystem Expansion
*   **60% Effort:** Micro-Repository Focus (e.g., `py-bench-playground`).
    *   Build out specialized standalone tools.
    *   Release v0.1.0 or minor feature branches.
*   **30% Effort:** External OSS Outbound Contributions.
    *   Follow up on pending PRs, execute requested changes.
*   **10% Effort:** Flagship Light Triage.
    *   Merge dependency bots, triage newly generated backlog issues.

## Metrics of Success
*   The contribution graph should display heterogeneous activity across 3-4 repositories consistently over a 30-day window.
*   Commits should be distributed evenly rather than appearing as massive 100-file dump spikes.
